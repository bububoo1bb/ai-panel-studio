# AI Panel Studio - Product Requirements Document

## 1. 产品概述

AI Panel Studio 是一款 AI 圆桌讨论 Web 应用。

用户输入任意讨论主题并指定参会专家人数后，系统通过大语言模型动态生成一名主持人与多名具有不同背景、职业和立场的专家。

主持人负责组织讨论，专家根据讨论上下文自主参与发言、补充、质疑与反驳。

讨论过程中，系统实时展示完整发言记录，并持续提炼当前形成的共识与分歧。

讨论结束后，由主持人完成自然收尾，并形成结构化的讨论总结。

## 2. 目标用户

### 2.1 知识工作者与研究人员

希望快速从多个不同视角分析一个复杂议题，发现潜在问题、分歧与新的思考方向。

### 2.2 产品与决策人员

希望在方案评审、战略讨论或决策前，通过 AI 多角色讨论发现盲点、检验假设并辅助决策。

### 2.3 教育与内容创作者

希望围绕某一主题获得结构化、多视角、可回放的讨论内容。

## 3. 核心用户流程

1. 用户进入首页。
2. 用户查看已有讨论，或创建新的讨论。
3. 用户输入讨论主题并指定专家人数。
4. 系统调用大语言模型生成主持人与专家阵容。
5. 用户查看并确认嘉宾信息。
6. 用户进入 AI 圆桌演播厅。
7. 主持人开场并推进讨论。
8. 专家根据当前上下文自主参与讨论。
9. 系统实时展示 Transcript。
10. 系统持续更新当前共识与分歧。
11. 主持人完成讨论收尾。
12. 用户查看最终讨论结果与总结。

## 4. MVP 功能需求

### 4.1 首页与讨论管理

首页展示当前存在的讨论。

用户可以：

- 查看已有讨论
- 加入已有讨论
- 创建新的讨论

不同讨论之间的数据与运行状态必须相互隔离。

### 4.2 创建讨论

用户可以：

- 输入讨论主题
- 指定专家人数

系统根据这些信息创建新的讨论。

### 4.3 AI 嘉宾生成

系统根据讨论主题动态生成：

- 1 名主持人
- 指定数量的专家

每位专家至少具有：

- 姓名
- 职业或 Title
- 立场或主要观点
- 专属视觉标识

用户确认嘉宾阵容后进入演播厅。

### 4.4 AI 圆桌讨论

主持人负责：

- 开场
- 提问
- 追问
- 串联不同观点
- 推进讨论
- 最终收尾

专家需要根据：

- 自身角色
- 自身立场
- 当前讨论上下文

自主决定是否参与当前讨论。

专家之间可以：

- 提出观点
- 补充
- 质疑
- 反驳

讨论不得采用固定机械轮流发言模式。

单次发言应保持简洁。

### 4.5 专家状态展示

每位专家具有独立的状态展示区域。

可以展示：

- 待机
- 准备发言
- 发言中
- 当前关注点
- 公开思考摘要

不得向用户展示模型完整的内部 Chain-of-Thought。

### 4.6 Transcript

系统实时展示讨论发言记录。

每条公开发言至少展示：

- 发言人姓名
- 职业或 Title
- 发言内容
- 与角色一致的视觉标识

Transcript 仅展示公开讨论内容。

内部调度事件不作为公开发言展示。

### 4.7 实时共识与分歧

系统在讨论进行过程中持续分析已有发言。

实时维护：

- 当前形成的共识
- 当前存在的主要分歧

该信息应随着讨论推进动态更新，而不是只在讨论结束后一次性生成。

### 4.8 讨论结束与总结

讨论结束时：

1. 主持人完成自然语言收尾。
2. 系统整理最终讨论结果。
3. 页面以用户可读形式展示总结。

不得直接向用户展示内部 JSON 数据。

### 4.9 多讨论隔离

系统支持多个讨论同时存在。

不同讨论之间的：

- 运行状态
- Transcript
- 专家信息
- 事件流
- 共识与分歧
- 最终结果

必须保持隔离。

## 5. 非功能需求

### 5.1 实时性

讨论过程应通过 SSE 或 WebSocket 等机制实时更新页面。

### 5.2 数据持久化

讨论及相关数据使用 SQLite 保存。

### 5.3 安全性

大模型 API Key：

- 仅允许在后端读取
- 使用环境变量保存
- 不得暴露到浏览器端

### 5.4 用户界面

- UI 使用中文
- 适配普通桌面浏览器
- 核心讨论区域应具有合理的独立滚动设计
- 用户能够清晰区分不同专家及其发言

## 6. MVP 成功标准

用户能够完整完成以下流程：

创建讨论
→ 生成并确认嘉宾
→ 进入演播厅
→ 观看 AI 多角色实时讨论
→ 查看 Transcript
→ 实时查看共识与分歧
→ 完成讨论
→ 查看最终总结

同时：

- 不同讨论的数据保持隔离
- 页面刷新后核心数据仍可恢复
- API Key 不暴露到前端
- 核心流程能够稳定运行

## 7. 当前版本暂不优先实现

在保证核心验收要求完成的前提下，以下功能不作为 MVP 优先目标：

- 用户注册与登录
- 社交分享
- 复杂权限系统
- 移动端 App
- 与核心圆桌讨论无关的数据分析功能
- 非必要的动画与视觉特效
